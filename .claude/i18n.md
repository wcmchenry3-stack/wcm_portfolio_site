# Localization (i18n) Standards

## Stack

- **Library:** i18next + react-i18next + i18next-http-backend
- **Locale files:** `public/locales/<locale>/<namespace>.json` (loaded at runtime)
- **Namespaces:** `common`, `home`, `resume`
- **Single source of truth for locales:** `src/i18n/locales.js` ‚Äî all other code derives from this

## Adding a New Locale

1. Add the locale object to `LOCALES` in `src/i18n/locales.js` (code, label, nativeLabel, flag, dir)
2. Create `public/locales/<code>/common.json`, `home.json`, `resume.json` with `__NEEDS_TRANSLATION__` as placeholder values for keys that need translation; preserve any values that should not be translated (e.g. `"navbar.brand": "Bill McHenry"`)
3. Run the translation script (see below)
4. No changes needed to `i18n.js` or the LanguageSwitcher ‚Äî they derive from `LOCALES`

## Translation Script

```bash
# Single locale + namespace
node scripts/translate.js --locale <code> --namespace <ns>

# Force retranslate all keys (not just placeholders)
node scripts/translate.js --locale <code> --namespace <ns> --force

# All locales, all namespaces (requires OPENAI_API_KEY)
for locale in fr-CA es hi ar zh ja ko pt he de nl ru; do
  for ns in common home resume; do
    node scripts/translate.js --locale $locale --namespace $ns --force
  done
done
```

Requires `OPENAI_API_KEY` in environment. Key is stored in `~/../.env.local` (outside the repo).

## Translation Philosophy

Translations are **not literal** ‚Äî they are marketing-first adaptations. The system prompt instructs the model to act as a bilingual product marketing expert writing for a global tech audience.

Key principles:
- **Adapt, don't translate** ‚Äî the goal is copy that sounds like it was written by a native-speaking tech copywriter
- **PM positioning** ‚Äî Bill is a Principal Product Manager; copy should emphasize product outcomes and user value, not just technical delivery
- **Hybrid terms are encouraged** ‚Äî use localized tech vocabulary where it feels native (e.g. `ý§°ý•áý§üý§æ-ý§°ý•çý§∞ý§øý§µý§®` in Hindi)
- **Headlines are punchy** ‚Äî concise, not full sentences
- **Elevate weak English** ‚Äî if the source copy is bland, the target should be stronger

If translations feel too literal or flat, update `buildSystemPrompt()` in `scripts/translate.js` and rerun with `--force`.

## Glossary & Protected Terms

`src/i18n/glossary.js` defines:
- **Protected terms** ‚Äî proper nouns and brand names that must never be translated (e.g. `Bill McHenry`, tool names)
- **Do-not-translate terms** ‚Äî passed to the model and validated post-translation

The script warns if a protected term is missing from the output.

## RTL Support

Locales with `dir: 'rtl'` (currently Arabic and Hebrew) are automatically handled:
- `RTL_LOCALES` set is exported from `locales.js`
- `useHtmlAttributes.js` sets `<html dir>` and `<html lang>` on locale change

## Tests

- `LanguageSwitcher.test.jsx` asserts `LOCALES.length` options are rendered ‚Äî automatically stays correct when locales are added
- No hardcoded locale counts in tests
